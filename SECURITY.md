# Безпека

## Підхід

Платформа використовує **принцип Керкгоффа**: безпека ґрунтується на секретності ключів, а не алгоритму. Алгоритми (AES, PBKDF2, Ed25519) публічні і перевірені — це нормально. Знати алгоритм недостатньо для розшифрування без ключів.

## Що захищається

| Обʼєкт | Захист |
|--------|--------|
| Runtime-пакети | Шифрування симетричним ключем, похідним від ліцензії + HWID + серверних даних |
| Bootstrap state | Fernet + на Windows додатково DPAPI |
| Локальні дані runtime | Шифрована SQLite (ключ з ліцензії + HWID + salt з маніфесту) |
| Маніфест | Підпис Ed25519, перевірка перед завантаженням |

## Компоненти для розшифрування

Щоб розшифрувати runtime-пакет, потрібні одночасно:

- **Ліцензійний ключ** — видається адміністратором, зберігається клієнтом
- **HWID** — ідентифікатор пристрою (привʼязка до машини)
- **Дані з маніфесту** — видаються сервером лише після успішної перевірки ліцензії

Без ліцензійного ключа та HWID відновити ключ шифрування неможливо.

## Криптографія

- **PBKDF2** — 600 000 ітерацій для похідного ключа (захист від brute-force на клієнті)
- **Fernet** — AES-128-CBC + HMAC для шифрування даних
- **Ed25519** — підпис маніфесту, перевірка цілісності

## Захист на сервері (brute-force)

Клієнтський PBKDF2 захищає лише локально збережені дані. Брутфорс через API — окремий вектор. Сервер додає:

| Захід | Опис |
|-------|------|
| **Rate limiting** | license/check: 15 запитів/хв на IP; license/info: 30/хв; admin login: 5/хв |
| **Затримка при помилці** | Невдала перевірка ліцензії — 3 с перед відповіддю; невдалий admin login — 5 с |

Разом: ~15 спроб/хв × 3 с = мінімум 45 с на 15 спроб. Мільйон спроб — роки. Brute-force через API стає непрактичним.

## Додаткові шари (Windows)

| Захід | Опис |
|-------|------|
| **DPAPI** | Bootstrap state шифрується через Windows DPAPI. Викрадений локальний кеш не розшифрується на іншій машині або під іншим користувачем Windows. |
| **Heartbeat** | Runtime періодично перевіряє ліцензію на сервері. Якщо ліцензію відключено — застосунок завершується. Дозволяє відкликати доступ без очікування перезапуску. |

## Ризики

| Сценарій | Захист |
|----------|--------|
| Хтось має тільки код платформи | Не може розшифрувати — немає ключів |
| Викрадений зашифрований пакет | Не розшифрується без ліцензії + HWID |
| Повний доступ до машини користувача | Ускладнено: DPAPI, heartbeat. Можливий перехоплення з памʼяті під час роботи. |
| Підміна клієнта | Потрібен доступ до машини; сервер перевіряє ліцензію при кожному запиті |

## Що захищати

- Ліцензійні ключі не повинні потрапляти в логи або публічні репозиторії
- Сервісні ключі підпису та адмін-доступу — тільки на сервері, не в клієнті
- Користувач не повинен запускати змінений або шкідливий клієнт
